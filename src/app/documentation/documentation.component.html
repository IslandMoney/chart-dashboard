Documentation

1. Project Description

The purpose of this Application is to display charts in dragable and resizable containers.
The user needs to be able to choose the name, chart type and chart data that will be displayed.
Using Spring Boot we get the data from an SQL server (listening on port:3306) and then displayed on port:8080 as a JSON Object.
Then, using ANGULAR HttpClientModule we map the data to an Object. Afterwards to multiple arrays with the respective data.

2. Technical Elements
- Architecture
- Technologies Used:
  • HTML
  • CSS & SCSS
  • SPRING BOOT
  • ANGULAR 7
  • SQL Database
- Database Description:
  • The database is a mockup weather data database created and populated via an SQL script.
         (it has data similar to actual weather data)
- Classes, Objects and More
  • SpringDataCrudJpa - the main class
  • WeatherRepository - the DAO
  • Weather - the Entity (gets data from database, defines fields, constructors, getters, setters & the toString)
  • WeatherController - the REST (exposes mappings for GET POST DELETE & PUT after the /api mapping)
- Endpoints
  • api/weathers/
  • api/weathers/:id
  • api/weathers/:id/temperature
  • api/weathers/:id/dailyHighTemp
  • api/weathers/:id/dailyHighTemp
  • api/weathers/:id/dewPoint
  • api/weathers/:id/windSpeed
  • api/weathers/:id/windDirection
  • api/weathers/:id/barometricPressure
  • api/weathers/:id/visibility
  • api/weathers/:id/sunrise
  • api/weathers/:id/sunset
- Information Flux
  • We listen to our database on port:3306 with Spring Boot, get the data, expose it on port:8080/weathers/api.
  • We map this data (which is a JSON object at this point) using HttpClientModule
         and save it on single arrays with their respective data.
  • Afterwards we show this data as charts on our dragable containers.
- Security & Access Control
  • Only accessible through port:4200
- Interface
  • After accesing the app the user is presented with a header and a grid(albeit with invisible lines).
  • If this grid is empty the user is promted to add a new chart with a card in the middle of the page.
  • On clicking "Add Chart", a grid item pops up promting the user to enter the chart name.
  • Afterwards, the user can select the type of chart & of data,
         right before the chart appears on a draggable and resizable container.
  • Since our middle box dissapears after adding the first grid item, the user can then add further grid items
         by clicking the "+" button at the bottom left of the page.
3. Functionalites of the System
- How Technical Elements Respond to Functionalites
  • In the case of our grid item the information is displayed and hidden using the angular *ngIf operator.
  • Using this operator we check if the needed information is given before hiding one element and showing another at its place.
  • Clicking on the back button at the top left of the grid items, changes the "hasChartSelection" & "hasDataSelection" values
         of the grid component respectively, thus allowing us to edit the grid item.
  • On clicking delete or the "x" button on the top right of the grid item, a function
         is executed to delete that item from the dashboard.

- How the Functionalities are Split into Modules
  • The AppComponent module will only have two Functions, that is to toggle the side menu (drawer)
         and to contain our router outlet.
  • The HomeComponent module will contain the Grid with all the grid item functionalities such as to give a name,
         chart type, chart data, delete etc.
  • The DocumentationComponent module will handle... you guessed it, the Documentation.
  • The DashboardService will hold our dashboard array of items.
  • The WeatherDataService will get our data by listening to "localhost:8080/weathers/api" and mapping it into
         an array of objects.
